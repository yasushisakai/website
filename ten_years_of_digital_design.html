日本で大学の学士を卒業して 10 年過ぎ、それ以降身をおいてる建築・都市計画 x デジタルデザインを自分をセンター・オブ・ユニバースに置き、振り返りたい。

1 グラスホッパー以前 -2008

基本的に Processing を使ってた。max msp も知ってた。2006 年にコスタス・テルジィディスの Algorithmic Architecture terzidis2006algorithmic (田中浩也さん監訳)って本が出る、巻末のプログラムは Maya の MEL だった記憶がある。松川昌平さんがそこの翻訳担当してた。

Kostas が SFC にきて４日間のワークショップを Processing でやる。(first processing sketch)

坂茂研では、坂さんの教育方針的にほとんどパソコン使わなかった。「コンピュータ使うのなんてアイデアなくなったおじいさんのすることだよ」って言われて驚愕した記憶がまだある。それはともかく、その教育方針に間違いはなかったと思う、教え方も多分アメリカのサイアーク譲り1だったのが良かったと思う。紙のシェルター、当時の総合政策の学部長の阿川弘之さんに壊されちゃったけど、阿川さんのおかげで初めて空間そのものに対する愛着の概念を教えてもらった。いや、法的に問題があったし、坂さんも意固地だったと思うけど、やり方が大学教育としてしょぼかった。説明を果たし、審議をもとに合意を形成し、コミュニティを考える格好のチャンスだったのに。レッシグ気取りでいうとアーキテクチャと（坂研という）規範が阿川弘之という法によって破壊されたのを感じた。

そんな坂研(半裸でとんかちトントン)とバランスをとるように池田靖史先生(論の構築)のところにいた。そうゆう人多かったけど、僕は特にプログラミングしてもいいんだって感じだった。

VectorWorks では VectorScript と行って Pascal ベースのスクリプト環境があって、個人的にどうも好きになれず。というか、建築デザインでプログラミングするやつなんて（今も）少なくて、道具や言語選びに将来的なインセンティブを見据えた考えを求めちゃいけなかった。はい、これ VectorScript だからやれみたいな。今の Graphisoft もそれを感じる。

VectorWorks そのものとしてはアトリエ系事務所で使われているからインターン行きたいしかっこいいからと覚える動機があったけど、操れるプログラミング言語はなんだろうが、（開発・販売元以外）誰も気にしなかった。今なら、多少なりともプログラミングを勉強するとしたら python でしょう。だってそれで就職/スタートアップできるもの。

ただし、AutoCAD が Lisp という言語がベースなのは今でもかっこいいと思う。eLisp とかたまに書くけど、変態だし、就職はできないと思う。

学部卒業間際に田中浩也先生がこれからどうするのって聞いてくれて、松川さんの授業取らなきゃモグリでしょと指摘され、松川さんの授業を履修、これも Processing。最近考えれば考えるほど SyncTokyo の Meme が与えた影響の事の大きさを感じるんですけど、それはまた別の投稿。そのまま翌年 TA になる。

卒業設計(2008)は Rhinoceros と vray でやったけど、多分 SFC、少なくとも建築クラスターの中では初めてだった。だからなにってことではないけど、歴史として記述しておきます。設計と構想とチーム統率に気を取られてスクリプトに手を出さなかった。ひと段落した後に作ってみたけど、当時は.NET で C#か VBScript のもどきの RhinoScript だった。

2 Grasshopper によるコンピュテーショナル・デザインの民主化 2008 - 2012

グラフィカル・プログラミング・ツールとしてグラスホッパーが出てきたのが 2007 年。そこからそれがメインストリームになったのはみなさん知っての通り。

他にいろいろベントレー社から同じコンセプトの仕組みが出てきたが、グラスホッパーのポータルを中心にしたコミュニティづくりに力を入れるという戦略によって淘汰される。

Rhinoceros Grasshopper が成功した最大の理由はこれだと思う。極論するとこれ一つだと思う。この前にもグラフィカル・プログラミング・ツールはあった。だけど、開発者を中心とした grasshopper3d.com の熱気はすごかった。

あのコミュニティがあったこそ、ディビット・ルッテンという開発者が神格化されたし、 Processing においても、Ben Fry がいるし（と John Maeda か）。よくコミュニティづくりってど根性ボトムアップ草の根かつみんなのためジャアスティス!!!なイメージだけど、むしろ中央（でも優しい）独裁者、というか帝国制を産む傾向があるよね。Linux の Linus 然り、python しかり。気持ち悪いオタクがそれぞれみんな見せ合って、褒め称え、時にはけなして、侃侃諤諤し、文化を築く。

僕も、たくさんスクリプトを書いたり、コンポーネントを書いたけど、コミュニティに対しての最大の貢献は…外聞を気にせず書こう。…伝説の質問をしたからだ。

当時 IronPython のコンポーネントが出たばかりでみんな試行錯誤していた。グラスホッパー内のデータ構造をその Python コンポーネント内でどのよう使うかを質問した。なぜ伝説となったかというと、その後 Python のコンポーネントのヘルプにこの質問にリンクが貼られたことになり、ごくごくたまにこっちでも、お前はあの質問の奴かと（特に優秀な学生に）聞かれることがあるからだ。

僕は物理エンジンのカンガルーや WebAPI とつなげてデータスクレイピングを可能にするプラグインを開発したわけでもない2けど質問一つが立派なコミュニティ還元の方法であると初めて感じた。日建設計にいたから、もちろん気軽に自分たちの仕事の内容を外に出せるわけでもなく。それでも、仲間に入れてくれるんだ的な。

これが民主化を肌に感じた瞬間だと思う。

ソフトでは sketch up がすごく盛り上がった、無料なのにすごいみたいな。松川さんと sketchup についてた ruby のインタプリタ使って動画作りのおてつだいした。実際は夏コスタスのところでインターンという名の下のただボストン松川家居候してただけ。ある一定以上モデルが複雑になると途端に辛くなるよね Sketch Up。田中先生も MediaLab にサバティカルしてた。めっちゃくちゃ励ましてくれた。

3 環境の時代 2012 - 2015

GH の民主化の影響によって色んなものがつながる。とりわけ、環境シミュレーションがでかかかった。それまでの不思議 Blob 形態をつくるのに責任を取れるのは Harvard GSD の心臓に毛が生えたひとにぎりのゲーリーもどきだったので、それ以外の私たち凡人は流線型の形の正当化と社会的説明責任のために環境を使った。海外（日本含め）この受けがよく、環境シミュレーション祭りになる。

「自然怖い」と「環境に優しい」はビジネス的にもアカデミアでもサスティナブルな話題だったのでお金・人件費を投入する動機にもなったのでしょう。反応拡散方程式で不思議な柄作りましょうよりはね。

建築の社会的影響はそれだけではなくてプロパガンダや技術と文化の交差を示したるものとしての効果もあるんですとあれだけブロッブアーキテクチャでそれを掲げてたのに、みんないきなり環境環境の大合唱でよくも恥を知らずに掌返すのかなんだこれって思ってた。

僕は日建に就職して、お世話になっていた山梨さんチーム、羽鳥さん角田さんがそのバランスがすごくて勉強になった。あとその視点から川合健二神。

構造設計やそのシミュレーションはその限りではなくて、本当に命かかってますから、ちゃんとやってほしいし、その蓄積があるからこそ守りと攻めのバランスがエンジニアングとして完成していた。

日建入ってすぐに構造の出力結果を Revit モデルに変換するプラグインを出水文二さんと作ってわかった。人って人件費という入力があってエネルギーを生み出す関数なんだと体感した。なんでこんなやばい言い方するかって、当時 BIM 到来によって 3D モデリングという工程が出てきて、え、なに、いきなり、それどこから捻出するのよっていう BIM 慎重派からの反応だった。

じゃあ、それ機械にやらせればいいんじゃないって、思って作ったんだけど、これは想定される人件費を削減するんだと。僕らのプログラムが文句ひとつ言わずに頑張って働くと周りの社員にお金が行くんだと。人件費を払って人から得られるエネルギーを３d モデルにする代わりに、Revit のホヤホヤ API 叩いて直接モデルを作ってるんだと思ったら最後、人とマシン混同してた。

それもあったり、デジタルデザインをやって適当にファサード散らせばいいみたいな雑な要求もなかったわけではなくで、デジタルデザインがいろんな建築家に使われて市民権を得た反面、悶々とする感じもありました。

人件費という数字に還元できちゃって人とプログラムの境がわからなくなって錯迷しつつ、ツールメーカーという称号をコスタスから受勲した以上環境シミュレーションを*ただ使う側*になりきることにプライドが許さず、（かと行ってナビエストークス方程式を勉強する時間も実務上ない）ghpython にも飽きたし。全く明後日の方向のウェブ界によく顔を出す。言語は Javascript, Python, PHP だった。そこでnigechizu(阿部さんと一緒に) とか、lmn architecture(角田さんと一緒に)とかを作る。

ここから徐々に集合知に意識が向いて行き、いよいよ差がわからなくなる。あと、ウェブやってると社会を考えることが多くなった。それで建築よりもひとつ大きくして都市に行ったのかな。

4 ソーシャル・ジャスティス・ウォリアー 2015-

超お世話になった日建設計を退職して MIT へ。

デジタルデザインという分野がライフゲーム不思議柄遊びから少し大人になって環境という社会問題に近い事に触れたからか、日本の場合は 3.11 の影響からか、アメリカの場合は社会とテクノロジーの差があまりにも顕著になったという事が規範化したからか、「世のためになるってどうゆう事だっけ」関連の問いが強くなった。最初の方はそれこそ社会善を追求するためにはという横断幕を貼っていた訳ですが、2017 年当たりから、いや、もしかして社会問題って全部テクノロジーのせいなんじゃない？というテクノロジーそのものに対する嫌疑が目立つようになりました。3

こういった技術やテクノロジー不信に対しての応戦としては、監視社会に対応するような技術をピックアップする人が多くなってきたと思います。でもそれを中心に回す暗号技術が一般の人たちに受け入れてもらえるのがすっごい難しい。

「頑張ってテクノロジーが助けるんです、世の中のためになるような大事なことなんです！、ププぷ、プライバシー大事です。」

といって出てくるのが、得体の知れない黒い物体4で、反応は

「いや良くわかりません、この黒い箱はなんですか。説明してください」

ってなって、それから、世の中誰も信じられない、桃源郷となるはずだったインターネットも変わってしまった、信じられるのはハッシュ関数で、ブロックチェーンとは何か、コンセンサス・アルゴリズムとはと高尚な説法の果て、怪しい黒い壺を売り、新しい社会への私たちと一緒に踏み出しましょう。と 6 時間ぶっ通しで謎の言語に対する講義を展開する。

といいながら、お金だまくらかす見たいな。こうゆうこともあって、またさらに、テクノロジー不信が募るという。いったんこうなると、標的認定されて、社会から信用されなくなっちゃう。

実際最近のアメリカの建築学科のマスター論文ってすごい miscelleneous「その他」っぽいとよく思う。工学部の中で建築+シビルという一番応用で社会的に直接的な影響があると思ってた自分たち5の立場が「face swap でポルノ動画の顔をフェミニストの顔に差し替えてやるぞ騒動」とか、「画像処理の顔認識処理において黒人女性検知の確度がひどいから自動運転になったら私たち引かれちゃう？」とか「ブロックチェーン」とかやられると社会の重心がどんどんバーチャルになっていくので、環境までは教えられたけど、正直どうすればいいのかわからない、みたいな感じ。
